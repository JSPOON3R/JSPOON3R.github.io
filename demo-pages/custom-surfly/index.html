Quick reference for how this Custom UI works.
What this demo is
This is a reference UI you can copy and adapt.
This is a Custom UI demo for Surfly. It shows the broad building blocks you typically need: (1) a small backend that securely stores your API key, (2) a front-end that calls that backend, and (3) Surfly’s JavaScript API to render sessions into your own iframe so you can add custom controls. This demo uses Google Apps Script as the backend because it’s quick to deploy, but you can replace it with any server.
High-level flow
  1. Configure a backend endpoint (Apps Script or your own server) with your Surfly API key.
  2. The UI calls the backend to create sessions, list history, and manage settings/agents.
  3. A webhook posts incoming calls to the backend, which stores a short queue for the UI to poll.
  4. When you answer/start, the UI uses the Surfly JavaScript API to join inside the Active Session iframe.
Why we use the JavaScript API for join links
The join link is used to start inside our own Active Session frame.
When we receive a leader/follower link (from “Start Session” or from the Queue webhook), we don’t just set iframe.src. Instead, we join via Surfly’s JavaScript API so the session is controllable in code (e.g. relocate/navigate, custom controls, consistent UI state, and a clean “Active session” area).
Surfly.session({}, LINK).startLeader("#activeFrame");
surflySession.relocate("https://example.com");

REST API reference
Your backend (Apps Script in this demo) calls Surfly’s REST API so your API key stays server-side.
POST /v2/sessions/
Create a new session.
GET /v2/sessions/
List sessions for History.
POST /v2/sessions/{session_id}/end/
End a session.

Queue via Webhook
Inbound calls are delivered to your backend.
The Queue tab polls your backend (getQueue). The backend stores incoming webhook payloads briefly so the UI can display new calls. When you click “Answer”, the UI joins using the follower link.
